# numeric-lang v0.2.0 — The Core

**The release the crate exists for.** v0.2.0 lands `parse`: numeric literals in every radix, with digit separators, and decimal floats parsed with correct rounding. This is the hard part of the roadmap, and it is not deferred. The whole public surface is one function and two types — documented, property-tested, and benchmarked.

## What is numeric-lang?

A parser for the numeric literals a language front-end sees. A lexer isolates the run of characters that makes up a number; `numeric-lang` turns that text into a typed value — an integer magnitude or a floating-point number — while getting the fiddly parts right: radix prefixes, `_` separators, and the exact decimal-to-`f64` conversion that a naive parser botches on the last bit.

## What's new in 0.2.0

### `parse` — one function, every literal form

```rust
use numeric_lang::{parse, Number};

assert_eq!(parse("42"),        Ok(Number::Int(42)));
assert_eq!(parse("0xFF"),      Ok(Number::Int(255)));
assert_eq!(parse("0o755"),     Ok(Number::Int(493)));
assert_eq!(parse("0b1010"),    Ok(Number::Int(10)));
assert_eq!(parse("1_000_000"), Ok(Number::Int(1_000_000)));
assert_eq!(parse("6.022e23"),  Ok(Number::Float(6.022e23)));
```

The radix comes from a `0b` / `0o` / `0x` prefix (case-insensitive), defaulting to decimal. A literal is a float when — and only when — it carries a `.` or a decimal exponent, so `10` is `Int(10)` and `10.0` is `Float(10.0)`. A `_` separator is accepted anywhere strictly between two digits. Integers hold an unsigned `u128` magnitude; a leading sign is rejected, because sign is the lexer's concern (a separate token), not part of a literal.

`parse` is **total**: every input either yields a `Number` or a `ParseNumericError`. It never panics.

### `Number` — the typed result

```rust
use numeric_lang::{parse, Number};

let n = parse("0x2a").unwrap();
assert!(n.is_int());
assert_eq!(n.as_u128(), Some(42));
assert_eq!(n.as_f64(),  42.0);

match parse("1.5").unwrap() {
    Number::Int(v)   => println!("int {v}"),
    Number::Float(v) => assert_eq!(v, 1.5),
}
```

`Int(u128)` or `Float(f64)`, with `is_int` / `is_float` predicates and `as_u128` / `as_f64` accessors. It is `Copy` and `PartialEq`, and deliberately not `Eq`/`Hash` because `f64` is not.

### `ParseNumericError`

`ParseNumericError` is a `#[non_exhaustive]` enum that says exactly why a literal was rejected, carrying the offending byte offset where one applies:

```rust
use numeric_lang::{parse, ParseNumericError};

assert_eq!(parse("0b1210"), Err(ParseNumericError::InvalidDigit { index: 4 }));
assert_eq!(parse("1__0"),   Err(ParseNumericError::MisplacedSeparator { index: 2 }));
assert_eq!(parse("-5"),     Err(ParseNumericError::UnexpectedChar { index: 0 }));
assert_eq!(parse("0x"),     Err(ParseNumericError::MissingDigits));
```

It implements `Display` and `core::error::Error`, so it drops straight into a `?`-based error chain.

### Correct rounding, without reinventing it

Correctly-rounded decimal-to-`f64` conversion is a subtle, solved problem, and the standard library already does it well. `numeric-lang` validates separator placement and hands the cleaned digits to the `f64` parser, inheriting its round-trip correctness. A float with no `_` is parsed in place with zero copies; only the separated case touches a fixed 512-byte stack buffer (hence the `FloatTooLong` variant, which no human-written literal approaches). The float path is allocation-free and `no_std`, and the hard values round-trip exactly:

```rust
use numeric_lang::{parse, Number};

for input in ["0.1", "0.2", "0.3", "2.2250738585072014e-308"] {
    let expected: f64 = input.parse().unwrap();
    assert_eq!(parse(input), Ok(Number::Float(expected)));
}
```

### Optional `serde` support

Behind the `serde` feature, `Number` derives `Serialize` / `Deserialize`.

### Tests and benchmarks

- `tests/proptests.rs` — a `proptest` suite for the contracts that make the parser trustworthy: any `u128` round-trips through every radix, any finite non-negative `f64` round-trips through its shortest decimal form bit-for-bit, a `_` inserted between digits is transparent, and parsing is total over arbitrary input.
- `tests/integration.rs` — the public surface exercised as a consumer sees it.
- `benches/bench.rs` — Criterion benchmarks for each parse path.

## Performance

Latest local Criterion means, Linux x86_64 (WSL2 Ubuntu), Rust stable, release build:

| Literal | Example | Time |
|---------|---------|-----:|
| Plain float | `6.02214076e23` | 10 ns |
| Decimal integer (10 digits) | `1234567890` | 12 ns |
| Float, 16 significant digits | `3.141592653589793` | 16 ns |
| Hex integer | `0xDEADBEEFCAFE` | 19 ns |
| Separated integer | `1_000_000_000_000` | 19 ns |
| Separated float | `1_000_000.000_001` | 33 ns |
| `u128::MAX` (39 digits) | 39-digit decimal | 50 ns |

Numbers vary by CPU; run `cargo bench` on your target for a baseline. No path allocates.

## Breaking changes

**None** that affect real code — v0.1.0 was a scaffold with no public API. Three structural repairs landed on the way: the manifest `keywords` / `categories` were unquoted bare identifiers (the crate did not parse as valid TOML) and are now proper string arrays; `clippy.toml` MSRV was aligned with `Cargo.toml` at 1.85; and the `deny.toml` header and a stale README link were corrected.
