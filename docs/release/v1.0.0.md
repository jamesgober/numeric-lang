# numeric-lang v1.0.0 — API Freeze

**The surface is now stable.** v1.0.0 freezes the public API introduced in 0.2.0 under Semantic Versioning. There are no code changes from 0.2.0 — this release marks the contract as fixed and completes the roadmap.

## What is numeric-lang?

A parser for the numeric literals a language front-end sees: integers in every radix (`0b` / `0o` / `0x` / decimal) with `_` digit separators, and decimal floats parsed with correct rounding. The whole public surface is one function and two types — [`parse`](https://docs.rs/numeric-lang/latest/numeric_lang/fn.parse.html), [`Number`](https://docs.rs/numeric-lang/latest/numeric_lang/enum.Number.html), and [`ParseNumericError`](https://docs.rs/numeric-lang/latest/numeric_lang/enum.ParseNumericError.html). It is `#![forbid(unsafe_code)]`, `no_std`, allocation-free, and never panics.

## The stability promise

As of 1.0.0 the public API is frozen. Within the `1.x` series:

- `parse`, `Number` (its variants and `is_int` / `is_float` / `as_u128` / `as_f64` methods), and `ParseNumericError` will not change in a breaking way. A breaking change means a new major version.
- The **accepted grammar** is part of the contract: a literal that parses today parses to the same `Number` in every `1.x`. New forms may be *accepted* in a minor release, but a parsing input will never start being rejected — or change its value — without a major bump.
- `ParseNumericError` is `#[non_exhaustive]`, so a newly distinguished failure is an additive minor change. Match it with a wildcard arm.
- The `serde` representation of a `Number` (an externally-tagged enum) is fixed within `1.x`.
- MSRV (Rust 1.85) is a compatibility surface: raising it is a documented minor change, never a patch.

Full details in [`docs/API.md`](https://github.com/jamesgober/numeric-lang/blob/main/docs/API.md#stability).

## Since 0.2.0

Nothing functional. 0.2.0 delivered the parser; 1.0.0 promises not to break it. If you are already on `numeric-lang = "0.2"`, moving to `"1"` is a no-op recompile.

## Performance

Unchanged from 0.2.0. Latest local Criterion means, Linux x86_64 (WSL2 Ubuntu), Rust stable, release build:

| Literal | Example | Time |
|---------|---------|-----:|
| Plain float | `6.02214076e23` | 10 ns |
| Decimal integer (10 digits) | `1234567890` | 12 ns |
| Float, 16 significant digits | `3.141592653589793` | 16 ns |
| Hex integer | `0xDEADBEEFCAFE` | 19 ns |
| Separated integer | `1_000_000_000_000` | 19 ns |
| Separated float | `1_000_000.000_001` | 33 ns |
| `u128::MAX` (39 digits) | 39-digit decimal | 50 ns |

Numbers vary by CPU; run `cargo bench` on your target for a baseline. No path allocates.

Run on Windows x86_64 with testing on Linux (WSL2 Ubuntu), Rust stable and the 1.85 MSRV; the same commands run in the CI matrix across Linux, macOS, and Windows:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo clippy --no-default-features -- -D warnings
cargo test
cargo test --all-features
cargo build --no-default-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo deny check
```

All green. Counts at this tag:

- Default features: 21 unit + 6 integration + 5 property + 7 doctests.
- `--all-features`: identical, plus the serde-gated derives compile clean.

## Installation

```toml
[dependencies]
numeric-lang = "1"

# With serde:
numeric-lang = { version = "1", features = ["serde"] }

# no_std:
numeric-lang = { version = "1", default-features = false }
```

MSRV: Rust 1.85.
